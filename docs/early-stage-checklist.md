## 1. MVP 범위 고정 (필수)

- [ ] 이번 릴리즈에서 "꼭 필요한 기능"만 문서로 고정했다.
- [ ] "이번엔 안 하는 기능"도 명시했다.
- [ ] 기능별 완료 기준(Definition of Done)을 적었다.

쉬운 설명:
- MVP는 "최소 기능 제품"입니다.
- 초보 프로젝트가 실패하는 가장 흔한 이유는 "기능을 계속 늘리는 것"입니다.
- 그래서 "이번에 할 것/안 할 것"을 먼저 확정해야 합니다.

권장 산출물:
- `docs/mvp-scope.md`
- 예시 항목:
  - 이번에 한다: 회원 조회, 파티 생성/조회, 기본 예외 응답
  - 이번에 안 한다: 관리자 화면, 고급 검색, 고도화 통계

완료 기준:
- 팀원(또는 미래의 나)이 문서만 보고 "이번 범위"를 바로 이해할 수 있다.

## 2. DB 마이그레이션 도구 도입 (강력 권장)

- [ ] Flyway 또는 Liquibase 중 1개를 도입했다.
- [ ] 첫 번째 스키마 파일을 버전 기반으로 만들었다.
- [ ] DB 변경은 SQL 파일(PR)에 남기는 규칙을 만들었다.

쉬운 설명:
- "DB 마이그레이션"은 DB 구조 변경 이력을 코드처럼 관리하는 방법입니다.
- 나중에 컬럼 추가/삭제 시 "누가 언제 뭘 바꿨는지" 추적이 쉬워집니다.

왜 지금 해야 하나요?
- 초기에 안 하면 나중에 DB 상태가 사람마다 달라져서 장애가 납니다.

완료 기준:
- 새 환경에서 앱 실행 시 마이그레이션이 자동으로 적용된다.

## 3. 환경 분리 정책 확정 (local/local-emulator/prod)

- [ ] `local`, `local-emulator`, `prod` 프로필 역할을 정의했다.
- [ ] 비밀값(DB 비밀번호, 토큰)을 코드에서 제거했다.
- [ ] 환경변수/Secrets로만 비밀값을 주입한다는 규칙을 문서화했다.

쉬운 설명:
- `local`: 프론트+백엔드 통합 개발용
- `local-emulator`: Firebase Auth Emulator 기반 백엔드 단독 인증 테스트용
- `prod`: 실제 서비스용
- 환경마다 설정이 다르기 때문에 섞이면 사고 납니다.

현재 프로젝트 권장:
- 공용 파일 `application.yaml`에는 환경변수 참조만 유지
- 프로필 파일(`application-local.yaml` 등)은 정책만 담고 Git으로 추적
- 실제 값(DB 비밀번호, Firebase 경로 등)은 `.env` 또는 Secrets로만 주입
- `local`은 실제 Firebase 기반 통합 테스트이므로 `FIREBASE_PROJECT_ID`와 서비스 계정 파일 경로가 필요
- `local-emulator`는 Firebase Auth Emulator 기반 백엔드 단독 테스트이므로 `FIREBASE_AUTH_EMULATOR_HOST`, `FIREBASE_PROJECT_ID`만 주로 사용하고, 자격증명 파일 경로는 비워 두는 것을 기본으로 함
- 두 로컬 프로필의 기본 DB는 `localhost:3306`으로 두고, 필요하면 `DB_URL`로 Docker MySQL(`3307`) 같은 다른 포트를 덮어쓴다
- `local-emulator`는 기본적으로 로컬 DB 스키마를 재생성하지 않도록 유지

완료 기준:
- 저장소 코드만 봐도 비밀번호/토큰 같은 민감 정보가 없다.

## 4. 테스트 최소 기준 정하기

- [ ] PR 머지 전 최소 테스트 기준을 정했다.
- [ ] 핵심 비즈니스 로직(Service) 테스트를 시작했다.
- [ ] "버그 재현 테스트 -> 수정" 습관을 문서화했다.

쉬운 설명:
- 테스트는 "미래의 나를 위한 안전장치"입니다.
- AI 에이전트와 같이 개발하면 특히 회귀 버그 방지용 테스트가 중요합니다.

초보자용 최소 기준(추천):
1. 핵심 시나리오 1개 이상 테스트
2. 예외 시나리오 1개 이상 테스트
3. `./gradlew build` 성공

완료 기준:
- 기능 수정 후 기존 기능이 깨졌는지 자동으로 확인 가능하다.

## 5. API/에러 계약 고정

- [ ] 공통 응답 형식(`ApiResponse`)을 고정했다.
- [ ] `ErrorCode`와 HTTP 상태 매핑 규칙을 정리했다.
- [ ] 문서(`api-specification.md`)와 코드가 일치하는지 검증했다.

쉬운 설명:
- "계약"은 프론트/백엔드가 약속한 규격입니다.
- 계약이 자주 흔들리면 프론트가 계속 깨집니다.

완료 기준:
- 같은 종류의 오류는 항상 같은 형식/코드로 내려온다.

## 6. 로컬 실행 재현성 확보

- [ ] 신규 개발자가 10분 내 실행 가능한 가이드를 작성했다.
- [ ] 필수 도구(Java, DB, Gradle) 버전을 명시했다.
- [ ] 가능하면 `docker-compose`로 DB를 쉽게 띄우게 했다.

쉬운 설명:
- "내 컴퓨터에서만 된다"는 상태를 없애는 작업입니다.

권장 산출물:
- `README.md`의 "로컬 실행" 섹션
- 예시 명령:
  - `set -a && source .env && set +a && SPRING_PROFILES_ACTIVE=local ./gradlew bootRun`
- 실무 추천:
  - MySQL/Redis만 Docker로 올리고 앱은 IntelliJ/`bootRun`으로 실행
  - 앱까지 Docker로 올릴 때는 코드 변경 후 `docker compose up -d --build`로 재빌드

완료 기준:
- 문서만 따라하면 다른 사람도 동일하게 서버를 띄울 수 있다.

## 7. 보안 기본선 설정

- [ ] 민감 정보 로그 출력 금지 규칙을 정했다.
- [ ] 인증/인가 책임 경계를 명시했다.
- [ ] 외부 공개 API와 인증 필요 API를 구분했다.

쉬운 설명:
- 인증(Authentication): "너 누구야?"
- 인가(Authorization): "이 작업 해도 돼?"

현재 프로젝트 맥락:
- Firebase ID Token 검증 기반
- 도메인 권한 판단은 서버에서 수행

완료 기준:
- "누가 어떤 API를 호출할 수 있는지"가 문서와 코드에 명확하다.

## 8. 관측성(Observability) 최소 세팅

- [ ] 헬스체크 엔드포인트를 준비했다.
- [ ] 요청/에러 로그 포맷을 통일했다.
- [ ] 장애 시 어디 로그를 보면 되는지 문서화했다.

쉬운 설명:
- 관측성은 "문제 생겼을 때 빠르게 찾기 위한 장치"입니다.
- 기능 구현만 하고 관측성을 안 하면 장애 원인 찾는 데 시간이 오래 걸립니다.

초보자용 최소 구성:
1. 서버 살아있는지 확인할 health endpoint
2. 에러 로그에 request path + errorCode 남기기
3. 운영/개발 로그 구분

완료 기준:
- 장애 재현 없이도 로그로 원인 후보를 좁힐 수 있다.

## 9. 운영 체크리스트(배포 전/후) 만들기

- [ ] 배포 전 체크리스트 작성
- [ ] 배포 후 체크리스트 작성
- [ ] 롤백 절차(문제 시 되돌리는 방법) 작성

쉬운 설명:
- 배포는 "코드를 올리는 행위"가 아니라 "서비스를 안전하게 바꾸는 절차"입니다.

권장 항목:
- 배포 전:
  - CI 통과 여부
  - DB 마이그레이션 검토
  - 환경변수 확인
- 배포 후:
  - 헬스체크
  - 핵심 API smoke test
  - 에러 로그 모니터링
- 롤백:
  - 이전 버전 재배포 방법
  - 비상 연락/판단 기준

완료 기준:
- 실제 장애 상황에서 문서만 보고 대응 가능한 수준
